# 论文写作支撑（完全依照当前项目实现）

## 1. 项目现状与可写作定位

### 1.1 项目类型
- 本项目属于“法律智能问答与辅助分析”的**工程型系统研究**，核心价值是：
  - 本地知识优先（案例+法条），约束大模型幻觉；
  - 法官修订标准注入，支持人在回路（Human-in-the-loop）；
  - 动态向量检索与静态映射融合（Hybrid）提升可解释性与覆盖度。

### 1.2 当前功能闭环（以代码为准）
- 主应用：`04.ai法律助手.py`
- 向量检索：`retriever.py`
- 法条匹配与可视化：`law_service.py`
- 法官修订依据（Hybrid）：`modification_service.py`
- 提示词约束：`prompts.py`
- 交互组件：`ui_components.py`
- 会话持久化：`session_manager.py`
- 评测脚本：
  - 阈值扫描：`evaluate_retrieval.py`
  - 论文评测（严格版）：`evaluate_paper_package.py`

### 1.3 数据资产（已核验）
- `resources/cases.json`：2001条刑事案例
- `resources/laws.json`：452条法条数据
- `resources/crime_patterns.json`：618个罪名映射
- `resources/case_index.faiss`：案例向量索引

## 2. 系统方法可写点（可直接写“方法章节”）

### 2.1 检索增强分析（RAG）
- 使用 `BAAI/bge-large-zh-v1.5` 编码案例事实。
- 建索引时做归一化向量，使用 `faiss.IndexFlatIP` 检索。
- 在线推理时支持相似度阈值筛选，并保留相似度分数用于可解释展示。
- 当检索为空时有降阈值重试逻辑，减少“误判为资料不足”。

### 2.2 法条回查机制
- 从“关联索引 + 用户输入”中抽取条号（含条/款）。
- 做中文数字与阿拉伯数字归一（如“第二百三十二条”与“第232条”对齐）。
- 输出法条原文摘要并提供命中调试信息（可选）。

### 2.3 法官修订标准注入
- 法官在操作台提交：
  - 拟修订结论；
  - 修订理由；
  - 核验证据要点。
- 系统将其固化到会话状态，并在后续轮次与本地知识同级纳入推理上下文。
- 提示词已明确：若法官标准与检索证据冲突，先列冲突点，再给保守结论。

### 2.4 Hybrid 映射（动态+静态）
- 动态层：`retriever.get_crime_info()`实时检索罪名相关案例并抽取事实/法条/案例。
- 静态层：`crime_patterns.json` 提供补全与兜底。
- 融合层：按“动态优先、静态补足、去重限长”得到最终修订依据。

## 3. 实验与结果支撑（当前可用）

### 3.1 现有评测产物
- 指标总文件：`paper_package/results/paper_metrics.json`
- 论文表格文件：
  - `paper_package/results/paper_metrics_table.csv`
  - `paper_package/results/paper_metrics_table.md`

### 3.2 当前已产出指标（严格版）
- 检索严格指标（示例一次运行）：
  - Hit@K
  - Precision@K
  - MRR@K
  - nDCG@K
  - 平均相似度、平均返回数量
- 映射模式对比：
  - dynamic / static / hybrid 的 fact/law/example coverage
  - 各模式平均案例规模与信息量

### 3.3 论文里如何解释这些数值
- 如果 `Hit@K` 很高而 `Precision@K` 约0.5：
  - 说明“命中能力强，但Top-K中仍有噪声项”，可通过阈值、重排优化。
- 若 `hybrid` 在 `avg_fact_count`、`avg_law_count` 上优于 `static`：
  - 可论证“动态抽取增强信息密度，静态映射提高稳定性”。

### 3.4 消融实验（已补充脚本）
- 脚本：`evaluate_ablation.py`
- 输出：
  - `paper_package/results/ablation_metrics.json`
  - `paper_package/results/ablation_table.md`
- 覆盖设置：
  - `full_hybrid`
  - `w/o static mapping`
  - `w/o dynamic retrieval`
  - `w/o judge standard`（离线近似）
- 说明：
  - 法官标准注入项由于本质是交互策略，脚本采用“案情+目标结论”构造查询做离线近似评估，论文中应明确该限制。
  - 在该近似评估中，建议优先报告以下更有区分度指标：
    - `top1_consistency`
    - `precision_target_at_k`
    - `mrr_target_at_k`
    - `law_support_rate`

## 5. 风险与答辩预案（必须写）

### 5.1 审稿高频质疑点
- 指标是否“过于乐观”（同库评测偏容易）。
- 标签质量（关键词抽取噪声）是否影响评测可信度。
- 是否真正减少了幻觉，还是只做了模板约束。

### 5.2 对应应对
- 在论文中明确“弱监督标签+同域评测”的局限。
- 增补少量人工标注评审（至少50条），报告一致性统计。
- 展示失败案例（误检索、法条误命中、结论冲突）并给出修正策略。
